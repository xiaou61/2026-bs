# 项目预览设计文档

## 目标

为仓库内真实存在的毕业设计项目建立一套可持续推进的预览体系：

- 从 `001` 开始逐个运行项目
- 生成多角色、多功能页面截图
- 为每个项目生成单独的详览页
- 为每 10 个项目生成一个分组索引页
- 在根目录 `readme.md` 中提供统一查看入口和说明

## 文档结构

预览文档采用三层结构：

1. `readme.md`
   - 只放总入口、查看说明、当前进度
2. `docs/previews/groups/<start>-<end>.md`
   - 每 10 个项目一组
   - 展示项目编号、项目名、状态、封面图、详览链接
3. `docs/previews/projects/<projectId>.md`
   - 单项目详览页
   - 按角色和功能页面组织 10 到 20 多张截图

截图资源统一存放在 `docs/previews/assets/<projectId>/`。

## 执行边界

本轮采用“启动与文档脚本化，截图通过浏览器工具完成”的方式：

- Python 脚本负责：
  - 扫描项目
  - 解析项目名和账号
  - 导入 SQL 初始化数据库
  - 启动后端和前端
  - 写入运行清单
  - 生成项目页、分组页、README 入口
- 浏览器工具负责：
  - 登录系统
  - 按角色遍历菜单
  - 对首页、列表、表单、详情、统计等页面截图

这个分工的原因是：

- 启动与记录天然适合脚本稳定执行
- 多页面截图需要实时观察页面状态，浏览器工具更稳
- 用户明确要求优先使用可操作的浏览器工具

## 分类规则

### A 类：`backend + frontend`

- 启动后端
- 启动前端开发服务
- 浏览器访问前端页面并截图

### B 类：`backend`

- 视作一体化项目或后端模板项目
- 直接访问后端页面截图

### C 类：`backend + miniprogram/miniapp`

- 优先找 Web 管理端或后端模板页
- 如无可访问 Web 页面，则记录为“小程序型项目，暂无线页截图”

### D 类：异常项目

- 启动失败
- 账号失效
- 页面为空
- 缺少可访问入口

这类项目必须记录失败原因，但不能阻塞后续项目推进。

## 截图组织方式

每个项目按角色组织截图，优先覆盖：

- 登录页
- 仪表盘
- 列表页
- 新增/编辑表单
- 详情页
- 审核/处理页
- 统计图表页
- 个人中心

默认目标：

- 每个项目至少 10 张
- 优先达到 15 到 20 张
- 如果系统角色较多，使用管理员、教师、学生或等价角色分别补足

## 运行策略

为减少项目间冲突，采用单项目串行执行：

- 后端默认使用项目约定端口，优先 `8080`
- 前端默认使用项目约定端口，优先 `3000`
- 每完成一个项目后及时停止对应进程

数据库采用本地 MySQL：

- 账号：`root`
- 密码：`1234`

Redis 依赖默认走本地 `127.0.0.1:6379`。

## 记录要求

每轮推进都需要留痕：

- 运行日志：`logs/project-preview/`
- 运行清单：`docs/previews/runtime/`
- 项目详览：`docs/previews/projects/`
- 分组索引：`docs/previews/groups/`
- 根入口：`readme.md`

## 当前实施顺序

1. 搭建预览流水线基础脚本
2. 完成 `001` 项目建库、启动、截图、文档、README 入口
3. 验证目录结构、命名和预览格式
4. 再按编号继续推进后续项目
