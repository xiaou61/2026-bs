# 孩童收养信息管理系统设计文档

## 目标
构建一个包含机构后台与申请人网页端的孩童收养信息管理系统，覆盖儿童信息展示、申请提交、审核评估、匹配审批、协议签订、回访跟踪和公告统计，满足毕设对业务完整性、角色协同和系统展示效果的要求。

## 设计结论
本项目采用后台管理端加申请人门户端的双端一体方案。后台由管理员与审核员使用，负责儿童档案维护、审核审批与运营统计；门户端由申请人使用，负责浏览信息、维护资料、提交收养申请和查询办理进度。后端统一提供 REST API，前端使用 Vue3 单页应用承载不同角色页面，通过 JWT 和 Redis 实现登录态校验。

## 方案比较

### 方案一：信息展示型
只实现儿童信息展示、在线申请和结果查询，开发速度快，但业务链条短，难以体现系统复杂度和管理价值，不适合作为“完整版”毕设项目。

### 方案二：业务闭环型
实现儿童档案、申请、审核、家访、匹配、协议、回访和统计全链路，覆盖主要业务节点，既能展示管理端能力，也能展示用户端交互，是完整性和实现成本之间的最佳平衡。

### 方案三：强监管型
在业务闭环型基础上继续加入多级审批、风险预警、黑名单和更细粒度审计，虽然更重，但会明显扩大开发范围，影响交付效率。

## 最终选择
采用方案二。该方案既能展示明确的角色分工和完整业务流程，又能控制表数量、接口数量和页面规模，适合当前仓库的项目风格和交付节奏。

## 系统架构
系统采用前后端分离架构。后端使用 Spring Boot 提供认证、业务处理和统一返回结果，数据访问层采用 MyBatis-Plus 提升 CRUD 开发效率，Redis 用于保存登录 Token 与会话信息，MySQL 存储业务数据。前端使用 Vue3、Element Plus、Pinia 和 Axios 组织页面与状态，路由按角色做权限控制。

系统逻辑上分为三层：认证与基础用户层、业务流程层、运营统计层。认证层处理登录注册和角色权限；业务层围绕儿童、申请人、申请、审核、匹配、协议、回访展开；运营层负责公告与首页看板。这样既能保持模块边界清晰，也方便后续直接按模块拆分后端控制器和前端页面。

## 角色与权限
- 管理员：管理儿童档案、公告、协议、统计和全局信息，可查看全部流程数据。
- 审核员：负责申请审核、家访评估、匹配审批和回访记录，重点处理业务流程节点。
- 申请人：维护个人资料、浏览可收养儿童、提交收养申请、上传材料、查看审批进度与公告。

权限设计保持简单稳定，不引入复杂 RBAC 表结构，直接基于用户角色控制菜单、接口和数据范围，更符合毕设项目成本控制。

## 核心数据模型
数据库控制在 12 张表左右。`sys_user` 负责认证与角色，`adopter_profile` 维护申请人档案，`child_profile` 和 `child_health_record` 存储儿童基本信息及健康记录，`adoption_application` 管理申请主流程，`application_material` 管理材料，`home_visit_record` 记录家访，`match_record` 记录匹配推荐，`approval_record` 记录审核审批意见，`adoption_agreement` 记录签约情况，`follow_up_record` 负责收养后回访，`system_notice` 提供公告展示。

儿童与申请之间采用“申请驱动”的方式组织流程。申请人先选择儿童提交申请，再经过材料审核、家访评估和匹配审批，最终进入协议登记与回访阶段。状态字段统一用数字枚举表示，前端负责映射成文案和标签颜色，减少数据库复杂度。

## 业务流程
申请人注册登录后完善个人档案，浏览可收养儿童并提交申请。管理员或审核员查看申请与材料，执行初审；通过后录入家访评估结果，再进行匹配审批；匹配通过后登记协议；协议完成后进入回访管理。申请人端可以查看个人申请状态、处理结果和公告信息。

系统中最重要的状态链是申请状态。建议设置为：待提交资料、待初审、待家访、待匹配、待签约、已完成、已驳回。这样既直观，又足以支撑页面展示、筛选和统计。

## 页面组成
后台页面包括登录、仪表盘、儿童档案、申请人资料、收养申请、审核评估、匹配审批、协议管理、回访管理、公告管理。申请人门户页面包括登录、注册、门户首页、儿童公开列表、儿童详情、我的资料、我的申请、申请进度、公告列表。

页面风格遵循仓库现有管理系统模式，后台采用“查询栏 + 表格 + 分页 + 弹窗”的统一结构，门户采用卡片列表与详情页结构，降低实现难度并保持一致性。

## 错误处理
后端统一使用 `Result` 返回结构和全局异常处理，业务异常统一抛出自定义异常。未登录、权限不足、参数错误、状态流转错误都返回明确消息。前端 Axios 拦截器负责统一弹出错误提示并处理登录失效。

## 测试策略
实现阶段优先为关键服务编写测试，重点覆盖登录注册、申请状态流转、审核通过驳回、匹配审批和统计聚合等核心行为。前端以接口联调和关键页面交互为主，至少保证登录、申请提交、审核流转和进度查询路径可验证。

## 交付边界
本项目不扩展文件真实上传服务、不接入短信通知、不实现第三方签章，材料与图片使用 URL 字段存储，协议使用文本与编号维护。这样可以保证项目完整但不过度膨胀，符合毕设演示与仓库现有项目节奏。
