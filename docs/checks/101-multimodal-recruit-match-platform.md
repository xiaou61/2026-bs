## 结论摘要

`101` 当前可以通过后端编译和前端打包，但仍不能判定为“通过”，主要阻断点在默认运行环境：

1. 默认配置强依赖本地 MySQL 数据库 `recruit_match_101` 和 Redis，首次启动后直接登录会返回 `500`。
2. SQL 需要手工执行 `101-backend/sql/init.sql` 才能创建演示库和默认账号，项目自身没有自举。
3. `PLAN.md` 明确写了“不执行编译验证”，与当前真实巡检要求不一致。
4. 后端没有自动化测试，`mvn test` 实际只是空测试通过。

## 已核对通过项

### 1. 构建链可通过

- 后端执行 `mvn.cmd test` 通过，但日志显示 `No tests to run`。
- 前端执行 `npm.cmd run build` 通过，存在大体积 chunk 警告但不阻断交付。

### 2. 登录返回已做密码脱敏

- `AuthService.login` 和 `AuthService.info` 均在返回前执行 `user.setPassword(null)`。

### 3. 主要角色边界基本符合实现预期

- 管理员看板 `/api/statistics/dashboard` 只允许 `ADMIN`。
- 候选人、简历、证书、面试反馈等核心服务层均实现了“本人数据”校验。
- 前端路由和侧边栏菜单与后端角色划分基本一致。

## 主要问题

### 101-1 默认环境不可直接运行

- `application.yml` 写死 MySQL `legal_case_101` 与 Redis `localhost:6379` 风格依赖中的本项目配置：`recruit_match_101`、Redis DB `4`。
- 在数据库未初始化时，四个默认账号登录都会返回 `Unknown database 'recruit_match_101'`。
- 数据库导入后，如果本机 Redis 未启动，登录又会返回 `Unable to connect to Redis`。

影响：

- 项目虽然“能编译”，但默认环境下无法完成最基本的登录验收。
- 演示前必须人工准备 MySQL 和 Redis，交付自解释性不足。

### 101-2 测试覆盖实际上为空

- `mvn.cmd test` 通过并不代表业务可用，当前后端没有任何测试源码。
- `PLAN.md` 仍把“不执行编译验证”写成完成标准，说明项目交付标准偏静态化，无法覆盖真实运行问题。

### 101-3 文档交付仍偏弱

- 运行依赖、导库步骤、Redis 前置条件没有在单项目文档中形成清晰启动说明。
- 这会导致复核者看到“默认账号”后仍无法直接登录。
