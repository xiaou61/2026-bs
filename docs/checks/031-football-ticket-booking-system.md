- 项目编号：`031`

- 检查日期：`2026-04-21`
- 当前状态：`已完成`
- 综合结论：`已完成默认环境自举、JDK 17 兼容复验、订票主链路修复、JWT 鉴权补齐、前端真实接口接入、后端集成测试补齐与浏览器复测；当前项目可在默认 H2 环境下通过测试并启动。`

## 2. 项目形态

- 后端目录：`031-backend`
- 前端目录：`031-frontend`
- PRD/说明文档：`项目目录下未发现独立 PRD 文件，本次按题目能力描述、数据库结构、前后端页面与接口实现进行核对`
- 后端技术栈：`Spring Boot 3.2 + Spring Data JPA + JWT + jOOQ（按需 codegen）`
- 前端技术栈：`HTML + CSS + JavaScript`

## 3. 主要问题与修复

### 3.1 后端修复

1. 将默认运行环境从强依赖本地 `MySQL + Redis + jOOQ codegen` 改为默认 `H2` 内存库可直接测试、直接启动，并保留 `application-mysql.yml` 供真实数据库切换。
2. 将 `jooq-codegen-maven` 挪入 `codegen` profile，避免默认 `mvn test` 因本地没有 MySQL 而直接失败。
3. 补充根级 `schema.sql`，替代原始 `sql/schema.sql`、`sql/data.sql` 中不适合默认嵌入库执行的初始化方式。
4. 新增 `DataInitializer`，自动初始化管理员、普通用户、球队、球场、未来赛事、票价分区与座位数据，解决默认环境无可用演示数据的问题。
5. 新增 `JwtInterceptor` 与 `AuthContext`，让 `/orders/**`、`/users/**` 等受保护接口真正执行登录鉴权。
6. 给 `User.password` 增加 `WRITE_ONLY` 序列化限制，修复用户信息接口暴露密码字段的问题。
7. 补齐票价查询与座位可用性查询接口，支持前端按真实赛事、真实分区拿到可售座位。
8. 重构下单逻辑，按真实票价计算订单金额，并在创建订单时生成锁定中的 `ticket` 明细。
9. 将订票闭环从“只改座位状态”改为“以 `ticket.status` 管理锁座、出票、取消”，避免球场座位被跨场次误判为永久售出。
10. 补齐余额支付逻辑，支付成功后将订单与票据状态推进到已支付/有效票，并回写用户余额。
11. 补齐电子票查看能力，使用户可以在订单详情中看到已支付门票信息。
12. 补齐取消订单时的库存释放逻辑，将对应票据改为取消态，恢复座位可售。
13. 新增后端集成测试，覆盖登录、赛事查询、座位查询、下单、支付、查看电子票与鉴权拦截。

### 3.2 前端修复

1. 修复首页赛事渲染逻辑，改为真实读取后端 upcoming 赛事数据，而不是只展示占位内容。
2. 修复登录页与注册页脚本，统一处理 HTMX/JWT 返回，确保登录后能正确写入本地令牌并跳转。
3. 重写订票页脚本，改为从真实票价接口、真实座位接口拉取数据，不再使用硬编码价位和伪造座位 ID。
4. 修复订单页逻辑，支持真实订单列表、余额支付、取消订单与电子门票查看。
5. 给静态页面补充空 `favicon`，消除浏览器控制台 `favicon.ico 404` 噪音。
6. 由后端 `WebMvcConfig` 直接挂载 `031-frontend` 静态目录，保证前端页面可直接经由后端统一访问与联调。

## 5. 当前实现与目标能力对照

### 5.1 已具备闭环

1. 用户注册、登录、个人资料查询与余额信息读取
2. 公开赛事列表、赛事详情与即将开始比赛展示
3. 按赛事查看票价分区与可售座位
4. 真实选座下单、金额计算与锁座
5. 余额支付、订单状态推进与电子票生成
6. 订单列表、订单取消与库存释放
7. 默认环境一键启动与后端集成测试

### 5.2 仍存在明显缺口

1. 支付能力仍为站内余额模拟支付，未接真实第三方支付网关。
2. 电子票当前以文本信息形式展示，尚未生成真实二维码图片或验票码图片资源。
3. 后台运营、通知消息、支付超时自动释放、赛事管理等扩展能力仍未完整落地。

## 6. 剩余风险

1. 项目默认环境已切到 `H2`；若后续切回 MySQL，还需要手动准备真实库配置，并在需要时显式启用 `codegen` profile。
2. 当前锁座后若用户长期不支付，尚未实现自动超时释放机制，仍可能产生短时库存占用。
3. 前端仍为手写静态页模式，复杂交互和管理端扩展成本较高。
4. 业务异常返回仍以项目内统一响应体为主，REST 语义化状态码覆盖不够彻底。
