- 项目名称：`居民智能健康管理平台`
- 检查日期：`2026-04-21`
- 当前状态：`已完成`
- 综合结论：`已完成默认环境自举、JDK 17 兼容复验、健康数据建表修复、角色权限加固、健康知识库模块落地、健康评估与预警闭环补齐、统计分析模块落地、自动化冒烟测试补齐与前后端构建/启动验证；当前项目可以在默认 H2 环境下通过测试并启动。`

## 2. 项目形态

- 后端目录：`030-backend`
- 前端目录：`030-frontend`
- PRD：`030-backend/PRD/030-智能健康管理平台-PRD.md`
- 后端技术栈：`Spring Boot 3.2 + Spring Data JPA + JWT + PostgreSQL/H2`
- 前端技术栈：`React 18 + Vite + Ant Design + Zustand`

## 3. 主要问题与修复

### 3.1 后端修复

1. 将默认运行环境从强依赖本地 `PostgreSQL + Redis` 改为 `H2` 文件库自举，补充 `application-postgresql.yml` 保留真库切换能力。
2. 关闭默认 `Redis repositories` 扫描，避免无实际使用场景下的多存储仓库噪音。
3. 修复 `schema.sql` 中无效的建库语句，避免后续误触发初始化时报错。
4. 修复 `HealthData.value` 对应数据库列名与 H2 保留字冲突的问题，将列名改为 `metric_value`。
5. 给 `User.password` 增加 `WRITE_ONLY` 序列化限制，修复注册与个人信息接口返回密码哈希的问题。
6. 新增默认数据初始化器，补齐可直接登录的患者、医生、管理员账号及基础患者/医生资料。
7. 限制注册角色仅允许 `PATIENT / DOCTOR`，避免通过注册接口直接创建管理员账号。
8. 给患者、医生、管理员相关接口补齐角色边界：
   - 患者档案与健康数据仅患者可访问
   - 医生资料维护仅医生可访问
   - 医生审核接口仅管理员可访问
   - 咨询详情、回复、评价按患者/医生/管理员角色分流
9. 新增后端冒烟测试，覆盖默认配置下的注册、登录、健康数据新增/查询以及管理员接口越权拦截。
10. 新增健康知识服务与控制器，补齐公开列表、公开详情、管理员列表、创建、编辑、上下线与删除接口。
11. 为健康知识模块补齐默认初始化数据，并在详情接口中回写浏览量，形成可演示的知识库闭环。
12. 新增健康评估服务与控制器，基于患者健康数据自动生成评估结果、风险因素、健康建议和预警消息。
13. 将健康数据新增/删除与评估服务联动，保证患者录入后可以自动获得最新评估结果。
14. 为预警消息增加未读统计、已读处理与简单去重策略，避免短时间内重复刷同类告警。
15. 新增统计分析服务与控制器，补齐管理员端和医生端真实统计接口。
16. 基于用户、医生、咨询、健康数据、知识库、预警等现有实体，聚合生成系统总览指标、近 7 天趋势和医生服务排行。

### 3.2 前端修复

1. 完成前端依赖安装并验证 `Vite` 构建通过。
2. 补充前端角色路由守卫，防止患者/医生/管理员跨角色直接访问其他后台页面。
3. 修复首页根路由跳转逻辑，改为按当前登录角色自动跳转到各自工作台。
4. 将“健康知识”占位页改为真实列表/详情页面，支持分类筛选、关键词检索与详情抽屉查看。
5. 新增管理员知识库管理页，支持知识内容新增、编辑、发布/下线与删除。
6. 新增患者“评估预警”页面，支持最新评估、历史评估、预警列表、未读统计与手动重新评估。
7. 在患者首页补充最新健康评分与未读预警数量展示，增强数据录入后的即时反馈。
8. 将管理员端和医生端 Dashboard 从静态占位数字改为真实统计大盘，补齐趋势图和服务量展示。

## 5. 当前实现与 PRD 对照

### 5.1 已具备闭环

1. 用户注册、登录、个人资料查看与更新
2. 患者健康档案创建与查看
3. 健康数据新增、按类型查询、按时间段查询、删除
4. 医生资料维护、医生列表查询与管理员审核
5. 在线咨询创建、患者查看、医生回复、患者评价
6. 健康知识公开列表、详情查看与管理员知识库管理
7. 健康评估自动生成、历史评估查看、异常预警列表与已读处理
8. 管理员端系统总览、近 7 天趋势和医生服务量排行
9. 医生端咨询统计、评分统计、近期咨询列表和服务趋势
10. 患者端、医生端、管理员端基础页面与路由骨架

### 5.2 仍存在明显缺口

1. 当前健康评估与预警采用规则引擎实现，尚不具备 PRD 设想中的深度疾病风险建模与个性化 AI 建议能力。
2. 当前统计分析已具备基础仪表盘能力，但还未达到 PRD 中“自定义报表生成”和导出级报表深度。
3. 医生排班、预约挂号、系统参数、操作日志、备份恢复、可穿戴设备对接均未实际落地。

## 6. 剩余风险

1. 前端生产构建仍有 `chunk size` 过大的告警，后续可做拆包优化。
2. 后端业务异常仍大量使用 `HTTP 200 + code/message` 返回，不是标准 REST 错误语义。
3. 默认环境已切到 H2；若后续切回 PostgreSQL，需要手动创建 `health_db` 并按需准备 Redis。
4. 健康知识、评估预警和统计分析已形成基础闭环，但通知推送、自定义报表和系统管理等高阶模块仍明显低于 PRD 描述范围。
