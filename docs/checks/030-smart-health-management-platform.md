# 030 居民智能健康管理平台检查报告

## 1. 检查结论

- 项目编号：`030`
- 项目名称：`居民智能健康管理平台`
- 检查日期：`2026-04-17`
- 当前状态：`已完成`
- 综合结论：`已完成默认环境自举、JDK 17 兼容复验、健康数据建表修复、角色权限加固、自动化冒烟测试补齐与前后端构建/启动验证；当前项目可以在默认 H2 环境下通过测试并启动。`

## 2. 项目形态

- 后端目录：`030-backend`
- 前端目录：`030-frontend`
- PRD：`030-backend/PRD/030-智能健康管理平台-PRD.md`
- 后端技术栈：`Spring Boot 3.2 + Spring Data JPA + JWT + PostgreSQL/H2`
- 前端技术栈：`React 18 + Vite + Ant Design + Zustand`

## 3. 主要问题与修复

### 3.1 后端修复

1. 将默认运行环境从强依赖本地 `PostgreSQL + Redis` 改为 `H2` 文件库自举，补充 `application-postgresql.yml` 保留真库切换能力。
2. 关闭默认 `Redis repositories` 扫描，避免无实际使用场景下的多存储仓库噪音。
3. 修复 `schema.sql` 中无效的建库语句，避免后续误触发初始化时报错。
4. 修复 `HealthData.value` 对应数据库列名与 H2 保留字冲突的问题，将列名改为 `metric_value`。
5. 给 `User.password` 增加 `WRITE_ONLY` 序列化限制，修复注册与个人信息接口返回密码哈希的问题。
6. 新增默认数据初始化器，补齐可直接登录的患者、医生、管理员账号及基础患者/医生资料。
7. 限制注册角色仅允许 `PATIENT / DOCTOR`，避免通过注册接口直接创建管理员账号。
8. 给患者、医生、管理员相关接口补齐角色边界：
   - 患者档案与健康数据仅患者可访问
   - 医生资料维护仅医生可访问
   - 医生审核接口仅管理员可访问
   - 咨询详情、回复、评价按患者/医生/管理员角色分流
9. 新增后端冒烟测试，覆盖默认配置下的注册、登录、健康数据新增/查询以及管理员接口越权拦截。

### 3.2 前端修复

1. 完成前端依赖安装并验证 `Vite` 构建通过。
2. 补充前端角色路由守卫，防止患者/医生/管理员跨角色直接访问其他后台页面。
3. 修复首页根路由跳转逻辑，改为按当前登录角色自动跳转到各自工作台。

## 5. 当前实现与 PRD 对照

### 5.1 已具备闭环

1. 用户注册、登录、个人资料查看与更新
2. 患者健康档案创建与查看
3. 健康数据新增、按类型查询、按时间段查询、删除
4. 医生资料维护、医生列表查询与管理员审核
5. 在线咨询创建、患者查看、医生回复、患者评价
6. 患者端、医生端、管理员端基础页面与路由骨架

### 5.2 仍存在明显缺口

1. PRD 中的健康评估、风险预警、健康建议、报表导出还没有形成完整控制器/服务闭环。
2. `HealthKnowledge` 实体存在，但后端没有对应控制器，前端“健康知识”仍是占位页。
3. 医生排班、预约挂号、系统参数、操作日志、备份恢复、可穿戴设备对接均未实际落地。
4. 统计分析与可视化能力目前仅有页面骨架，没有达到 PRD 描述的完整报表深度。

## 6. 剩余风险

1. 前端生产构建仍有 `chunk size` 过大的告警，后续可做拆包优化。
2. 后端业务异常仍大量使用 `HTTP 200 + code/message` 返回，不是标准 REST 错误语义。
3. 默认环境已切到 H2；若后续切回 PostgreSQL，需要手动创建 `health_db` 并按需准备 Redis。
4. 路由与接口的角色边界已补齐，但前端各角色页面的功能深度仍明显低于 PRD 描述范围。
